## What is KernelCMS?

**KernelCMS is an open-source (MIT), TypeScript-native, config-as-code headless CMS.** You model your
content in a single `kernel.config.ts` and get a typed content engine, an auto-generated REST and
GraphQL API, a typed in-process Local API, a polished React admin panel, and a CLI. Database, storage,
auth, email, cache, and search are all swappable adapters.

The guiding rule: **heavy or opinionated dependencies live behind optional adapters, never in the
core.** The default database is SQLite via Node's built-in `node:sqlite` (zero native dependencies),
so a fresh project boots in seconds. KernelCMS runs on web standards (`Request → Response`) with **no
framework lock-in** — self-hosted on a single container, or embedded in any framework.

What makes it different:

- **Add a backend to any site.** Give any website — even one generated by AI with v0, Lovable, Bolt, or
  Cursor — a database, an API, and a no-code admin, without changing the design.
- **Code stays code, content stays editable.** Developers own the model in typed config (versioned in
  git); editors change pages, copy, and media in a no-code admin with live preview and a page builder.
- **Three APIs from one model.** REST, GraphQL, and a typed Local API, all enforcing the same
  deny-by-default access rules.
- **Migrations you trust.** Diff-based, additive, deterministic — preview every change before it runs.

> This repository is the **website** for KernelCMS, not the CMS engine itself. The engine lives in the
> main [KernelCMS](https://github.com/RobbinPilhede/KernelCMS) repository.

## This project

A fast, content-rich site with a landing page, full documentation, guides, and a comparison blog —
designed to mirror the KernelCMS admin's visual language (monochrome, the animated radiant-star mark,
aurora + pixel-sweep background, glass surfaces).

### Tech stack

| Concern | Choice |
| --- | --- |
| Routing | [TanStack Router](https://tanstack.com/router) (file-based routes, type-safe) |
| Build / dev | Vite |
| UI | React 19 |
| Language | TypeScript (strict) |
| Content | Typed data modules (no external CMS — it's a static marketing site) |

### Project structure

```
public/                 Static assets served as-is
  brand/                Logo, favicon (white box + black star)
  og/                   1200×630 Open Graph cards (per comparison + default)
  assets/video/         Generated demo videos (demo.webm, walkthrough.webm)
  robots.txt, sitemap.xml, llms.txt, llms-full.txt
src/
  routes/               File-based routes
    __root.tsx          Root layout: background, persistent header + footer, ⌘K palette
    index.tsx           Home (hero, demo video, steps, code showcase, comparison)
    docs.index.tsx      Redirects to /docs/introduction
    docs.$slug.tsx      Doc page (sidebar nav + content + on-this-page TOC)
    guides.index.tsx    Guides grid (featured + cards)
    guides.$slug.tsx    Guide reader
    blog.index.tsx      Blog grid (featured + cards)
    blog.$slug.tsx      Article reader (TL;DR, FAQ, sources, cover art)
  content.ts            All content: docs, guides, comparison articles + SEO metadata
  icons.ts              Inline SVG icon set (feather-style)
  lib.ts                Syntax highlighter, cover-art SVG, brand mark, theme helpers
  ui.tsx                Components: Topbar, Footer, ThemeToggle, CommandPalette, DemoPlayer, Prose…
  views.tsx            Page views: DocPage, GuidesIndex, GuideReader, BlogIndex, ArticleReader, Card
  styles.css            Hand-tuned design system (tokens, animations, syntax + prose styles)
tools/                  Build-time generators (run with Node)
  build-seo.mjs         Static crawlable mirror + sitemap + llms.txt / llms-full.txt
  build-og.mjs          Renders the 1200×630 OG card PNGs
  demo-scene.html       Animated demo "scene" (AI chat → admin → live site)
  record-demo.mjs       Records the scene to demo.webm / walkthrough.webm
```

## Development

```bash
pnpm install        # install dependencies
pnpm dev            # dev server at http://localhost:3000
pnpm build          # production build to dist/
pnpm preview        # preview the production build
```

The TanStack Router plugin generates `src/routeTree.gen.ts` automatically on dev and build.

### Editing content

All copy lives in **`src/content.ts`** — each doc, guide, and article is a plain object with a
`slug`, metadata, and an HTML string. Add or edit an entry and it automatically appears in the
sidebar, the ⌘K command palette, and generates its own on-this-page table of contents from
`<h2 id="…">` headings. Per-page SEO titles and descriptions live in the same file's `SEO` map.

### Demo videos

`public/assets/video/demo.webm` and `walkthrough.webm` are **generated** from an animated on-brand
scene. To regenerate (with the preview server running):

```bash
node tools/record-demo.mjs       # records both .webm files from tools/demo-scene.html
```

Prefer real screen captures of the admin? Drop your own `demo.mp4` / `walkthrough.mp4` into
`public/assets/video/` — the player lists `.webm` first, then `.mp4`.

### SEO assets

`public/robots.txt`, `sitemap.xml`, `llms.txt`, and `llms-full.txt` welcome AI crawlers and provide a
machine-readable content index ([llmstxt.org](https://llmstxt.org)). The `tools/build-seo.mjs` and
`tools/build-og.mjs` generators produce the OG cards and the markdown index; see **Roadmap** for the
crawlable static mirror.

## Deployment

It builds to a static `dist/`, so it deploys anywhere:

```bash
pnpm build
# deploy dist/ to Cloudflare Pages / Vercel / Netlify / any static host
```

Set the production domain to **kernelcms.com**. After deploying, submit `sitemap.xml` to Google Search
Console and Bing Webmaster Tools.

## Roadmap

- **Prerendering / SSG.** This is currently a client-rendered SPA. To preserve full crawlability (the
  reason the static SEO mirror existed), add prerendering — either via TanStack Start's SSG or by
  pointing `tools/build-seo.mjs` at `src/content.ts` to emit per-page static HTML at build time.
- **Type-tighten** the ported `content.ts` / `lib.ts` / `ui.tsx` (currently `@ts-nocheck` on the
  ported-from-JS modules) into fully type-checked TypeScript.
